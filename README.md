## 📖 概要

**P2P Drop** は、ブラウザ間で**直接**ファイルを送受信する、**サーバーレス**かつ**エンドツーエンド暗号化**された一時ファイル共有アプリです。

- 🔐 **AES-GCM 256bit** で各チャンクを暗号化、鍵は URL ハッシュ（`#k=...`）に格納されサーバーに送信されません
- 🌐 **WebRTC（PeerJS）** で送信者と受信者のブラウザを直接接続
- 🔁 **双方向送信対応（v4.2）** — 一度繋いだ接続で何度でも往復可能、ファイル交換チャット風に使える
- 📁 **1つの HTML ファイル**で完結。ビルド・サーバー構築不要
- ⏱ **自動消去**機能で、設定時間が来たら通信を遮断し、ファイル参照を破棄

外部クラウドにファイルが残らないため、機密性の高い資料・私的な写真・大容量動画の**使い捨て共有**に最適です。

|------|------|
| **🔒 エンドツーエンド暗号化** | AES-GCM 256bit、チャンクごとに新しい IV を生成。鍵は URL ハッシュ（`#`以降）に格納され、シグナリングサーバーにも一切送信されません |

## 🚀 使い方

### 📤 送信者（ファイルを渡す側）

```
1. index.html をブラウザで開く
   ↓
2. ファイルをドラッグ&ドロップ or クリックで選択
   （複数ファイルは自動で ZIP に圧縮されます）
   ↓
3. 有効期限（5分〜3時間）を選択
   ↓
4. 大容量モード（StreamSaver）を必要に応じて ON
   ↓
5. 「暗号化して共有リンクを作成」をクリック
   ↓
6. 自動的に以下が実行されます:
   ・SHA-256 ハッシュ計算
   ・AES-GCM 256bit 鍵を生成
   ・Wake Lock 取得(スマホ画面ロック抑止)
   ・PeerJS で Peer ID 取得
   ↓
7. 生成された QR コードまたは URL を受信者に送る
   ・QR スキャン → 最速
   ・URL コピー → メッセンジャーで送信
   ↓
8. 送信完了 → 接続を維持したまま「双方向モード」に突入
   ・相手からの返送を待機
   ・「もう1つファイルを送る」で追加送信可能
```

> ⚠️ **重要**: 受信者がダウンロードを完了するまで、絶対にこのタブを閉じないでください。タブを閉じると共有が即停止します。

### 📥 受信者（ファイルを受け取る側）

```
1. 送信者から受け取ったリンクをブラウザで開く
   (QR コードをスマホでスキャンしてもOK)
   ↓
2. 自動的に以下が実行されます:
   ・URL ハッシュから暗号化鍵を取り出し
   ・送信者と P2P 接続
   ・ファイル情報（メタデータ）を受信
   ・もし切断したら最大3回まで自動再接続
   ↓
3. プログレスバー、転送速度、残り時間を確認しながら待機
   ↓
4. 受信完了 → SHA-256 検証 → 整合性バッジ表示
   ・🟢 整合性検証 OK
   ・🔴 ハッシュ不一致（破損）
   ・⚪ 検証スキップ（StreamSaver / 500MB超）
   ↓
5. 「デバイスに保存」ボタンでダウンロード
   または「相手にファイルを返送する」で
   → 同じ接続でそのまま逆方向にファイル送信可能！
```

## 🔁 双方向送信モード

P2P Drop **v4.2** から、初回の送受信が終わっても接続を維持し、**同じ接続でファイルを往復**できるようになりました。新しいリンクを発行する必要はありません。

### 動作の流れ

```
┌─────────────────────────────────────────────────────────┐
│  [Aさん]              [PeerJS シグナリング]      [Bさん]  │
├─────────────────────────────────────────────────────────┤
│  共有リンク発行 ───────→  PeerID 登録                    │
│                                                          │
│  ◆ ファイル選択                                          │
│  ◆ AES-GCM 鍵生成 → URL #k=... に格納                    │
│                                                          │
│  リンクを共有 ──────────────────────────────→  リンクで開く │
│                                                          │
│  ◀── 暗号化チャンク送信 ──→                              │
│  ◀── ...（チャンク往復）── →                             │
│  ◀── EOF ─────────────────→                              │
│                                                          │
│  ✅ 送信完了                                  ✅ 受信完了 │
│                                                          │
│  💎 接続維持 + 鍵を保持したまま「双方向モード」へ突入    │
│                                                          │
│  [Aさん] 待機中 ←─────────────────────  「返送する」を押す │
│                                       ファイル選択       │
│                                       既存鍵で暗号化     │
│                                       ◀── チャンク送信 ──│
│                                                          │
│  ✅ Aさんが自動的に受信モードへ切替                      │
│  ✅ 受信完了                            ✅ 送信完了      │
│                                                          │
│  💎 また双方向モードへ → A↔B 無制限に往復可能            │
└─────────────────────────────────────────────────────────┘
```

### 主な特徴

| 項目 | 内容 |
|------|------|
| **同じ接続を再利用** | 切断・再接続不要、シグナリングサーバーを再経由しない |
| **同じ暗号化鍵を再利用** | 新たな鍵交換が不要、URL の再発行も不要 |
| **役割の自動切替** | 送信者は送信完了後に自動で受信待機、相手からデータが来れば即受信開始 |
| **連続送信対応** | 「もう1つ送る」ボタンで何度でもファイル追加送信 |
| **タブを閉じれば全停止** | プライバシーは維持。両者がタブを閉じた時点で全データ破棄 |

### 双方向モードを使う方法

**受信者側から「返送」する**
1. 受信完了画面で **「相手にファイルを返送する」** ボタンをクリック
2. ファイル選択画面に切替（紫色の「双方向モード」バナーが表示）
3. ファイルを選んで **「暗号化して送信」**
4. 送信者側で自動的に受信モードに切替、ダウンロード可能

**送信者側から「もう1つ送る」**
1. 送信完了後の画面で **「もう1つファイルを送る」** ボタンをクリック
2. 同じ接続でファイル選択画面に戻り、追加で送信できる

## ⚠️ 動作環境とホスティング

このアプリは URL パラメータ（`?peer=...`）と URL ハッシュ（`#k=...`）を利用します。そのため、ローカルファイル（`file:///...`）として開いた状態では**別端末との共有はできません**。

### ✅ 方法1: GitHub Pages で公開（推奨）

<details>
<summary><b>📘 セットアップ手順</b></summary>

1. GitHub で新しいリポジトリを作成
2. `index.html` をアップロード
3. **Settings → Pages** へ移動
4. **Source** を `Deploy from a branch` に設定
5. **Branch** を `main` (root) に選択して **Save**
6. 数分後、`https://<username>.github.io/<repo>/` でアクセス可能

</details>

### ✅ 方法2: Vercel / Netlify / Cloudflare Pages

<details>
<summary><b>📗 セットアップ手順</b></summary>

1. [Vercel](https://vercel.com/) / [Netlify](https://www.netlify.com/) / [Cloudflare Pages](https://pages.cloudflare.com/) にサインアップ
2. 「New Project」または「Import from Git」を選択
3. リポジトリをインポート（または `index.html` をドラッグ&ドロップ）
4. デプロイ完了後、発行された HTTPS URL でアクセス

</details>

### ✅ 方法3: ローカルサーバー（同一ネットワーク内のみ）

```bash

python -m http.server 8000

# Node.js の場合

# PHP の場合
php -S 0.0.0.0:8000
```

その後、`http://<あなたのローカルIPアドレス>:8000/` で別端末からアクセス。

> 💡 **Tips**: VS Code 拡張機能「Live Server」を使うと簡単です。

### 🔐 大容量モード（StreamSaver）の追加要件

StreamSaver は Service Worker を用いた逐次保存機能で、以下の環境が必須です：

- ✅ **HTTPS 環境**（HTTP / file:// では動作しません）
- ✅ **Service Worker 対応ブラウザ**（モダンブラウザはすべて対応）
- ✅ **サードパーティ Cookie 許可**（ブロックしているとMITMページが開けない）

## 🛑 制限事項

| 項目 | 内容 |
|------|------|
| **メモリ使用量（通常モード）** | 受信側はブラウザ RAM 上でファイルを結合します。スマホで数 GB は厳しい場合あり |
| **メモリ使用量（StreamSaver モード）** | 数 GB〜数十 GB の超大容量も対応可能（HTTPS環境のみ） |
| **SHA-256 検証** | 500 MB 以下のファイルのみ。それ以上は計算をスキップ（メモリ保護） |
| **同時受信者** | 1対1のみ。2人目以降の接続は自動拒否 |
| **タブを閉じる動作** | どちらかがタブを閉じると接続終了、双方向モードも解除 |
| **NAT 越え** | 厳しい NAT 環境では P2P 直結失敗の可能性（将来的に TURN サーバー対応予定）|
| **再接続** | 同一セッション内で最大 3 回。タブを閉じての再開は非対応 |

## 🔐 セキュリティ設計

### 暗号化フロー

```
[送信側]
  1. AES-GCM 256bit 鍵を crypto.subtle.generateKey() で生成
  2. ファイルを 64 KB チャンクに分割
  3. 各チャンクに対して:
       ・ 12-byte ランダム IV を生成
       ・ AES-GCM で暗号化（出力: IV + 暗号文 + 16-byte 認証タグ）
  4. 鍵を base64url エンコード → URL ハッシュ `#k=...` に格納
  5. シグナリングサーバー(PeerJS)には Peer ID のみ送信、鍵は一切渡らない

           ┌──────────────────────────────────┐
           │ URL: ?peer=abc#k=<256bitキー>     │
           │       └────────┬────────┘        │
           │     サーバー    │ ブラウザ内のみ   │
           │     に届く     │ ハッシュは送信なし│
           └──────────────────────────────────┘

[受信側]
  1. URL ハッシュから鍵を抽出 → crypto.subtle.importKey()
  2. 受信チャンクの先頭 12 byte を IV、残りを暗号文として復号
  3. 全チャンク受信後、SHA-256 で整合性検証
  4. 整合性バッジで結果をユーザーに表示

[双方向モード時]
  ・ 同じ AES-GCM 鍵をメモリに保持しつづける
  ・ 返送時も同じ鍵で暗号化/復号
  ・ IV はチャンクごとに毎回ランダム生成(再利用しない)
  ・ 接続が切れるまで鍵を保持、切断時に破棄
```

### 脅威モデルと防御

| 脅威 | 防御手段 |
|------|---------|
| シグナリングサーバーの盗聴 | 🔒 鍵を URL ハッシュに格納（サーバーに送信されない） |
| ネットワーク経路上の中間者攻撃 | 🔒 WebRTC DTLS + AES-GCM の二重暗号化 |
| ファイル破損・改ざん | ✅ SHA-256 ハッシュ検証 + AES-GCM 認証タグ |
| リプレイ攻撃 | 🎲 チャンクごとにランダム IV を生成 |
| 鍵の URL 漏洩 | ⏱ 鍵はセッションごとに使い捨て、有効期限あり（5分〜3時間） |
| 復号エラー連発による DoS | 🛑 3回失敗で自動中断（無限ループ防止） |
| 複数受信者による状態破壊 | 🚫 既接続中は新規接続を自動拒否 |
| 双方向モード時の鍵漏洩 | 🔄 同一セッション内でのみ鍵を再利用、タブ閉鎖で完全破棄 |

### 鍵がサーバーに渡らない仕組み

URL: https://example.com/p2p-drop/?peer=ABC123#k=Xq8Y...

## ❓ FAQ

<details>
<summary><b>Q. 本当にサーバーにファイルは保存されないの？</b></summary>

A. はい。ファイルデータは送信者→受信者へ直接 WebRTC で転送されます。P2P 接続を確立するためのシグナリング処理だけは PeerJS の公開サーバーを経由しますが、サーバーは接続情報の橋渡しをするのみで、ファイル本体は通りません。さらに、**鍵は URL ハッシュ部に格納されておりサーバーには一切送信されない**ため、シグナリングサーバーの管理者でもファイルを復号できません。

</details>

<details>
<summary><b>Q. 通信は暗号化されている？</b></summary>

A. **2重の暗号化**で保護されています:

1. **WebRTC DTLS 層**: 通信経路は WebRTC の DTLS で自動暗号化
2. **アプリ層 E2EE（v4 新規）**: AES-GCM 256bit でチャンクごとに暗号化。鍵は URL ハッシュ（`#k=...`）に格納され、**シグナリングサーバーにも送信されません**

つまり、PeerJS の公開サーバーも、ネットワーク上の中間者も、ファイルの中身を見ることはできません。鍵を知らない限り復号不可能です。

</details>

<details>
<summary><b>Q. 双方向送信ではどうやって鍵を共有してるの？</b></summary>

A. **初回の送信時に生成した同じ AES-GCM 鍵を使い回しています**。具体的には:

1. 送信者が AES-GCM 256bit 鍵を生成し、URL ハッシュ（`#k=...`）に格納
2. 受信者は URL を開いたタイミングで鍵を取得・インポート
3. 受信完了後も、両者のブラウザのメモリに鍵を保持しつづける
4. 受信者が「返送する」を選んだ時、**同じ鍵で逆方向のファイルを暗号化**
5. 送信者側も同じ鍵で復号できる

**安全性**: 鍵はセッション中のみメモリに保持され、タブを閉じれば完全に破棄されます。新たな鍵交換が不要なため、シグナリングサーバーへの追加情報送信もありません。IVはチャンクごとに毎回ランダム生成するため、同じ鍵を使い回しても暗号文は安全です（AES-GCM の仕様）。

</details>

<details>
<summary><b>Q. 鍵はどうやって受信者に渡る？</b></summary>

A. 共有 URL の「ハッシュ部」（`#` 以降の部分）に鍵が含まれています。ブラウザはハッシュ部をサーバーに送信しないため、以下のような安全性が保たれます:

- ✅ PeerJS シグナリングサーバーには Peer ID しか送られない
- ✅ ホスティングサーバー（GitHub Pages 等）にも鍵は送信されない
- ✅ アクセスログにも鍵は記録されない

ただし、URL 自体をチャットアプリやメールに貼る際は、その手段のサーバーに URL 全体が渡るため、**エンドツーエンドで信頼できるチャットサービスの使用を推奨**します。

</details>

<details>
<summary><b>Q. 双方向モードで何回までファイルを往復できる？</b></summary>

A. **制限はありません**。接続が維持されている限り、A→B→A→B... と何度でもファイルをやり取りできます。1セッション中の有効期限（5分〜3時間）内であれば、新しいリンクを発行する必要もなく、追加のシグナリングサーバーアクセスもありません。

事実上、軽量な**ファイル交換チャットツール**として使えます。

</details>

<details>
<summary><b>Q. ファイルサイズの上限は？</b></summary>

A. モード別に以下が目安です：

| モード | 上限 | 備考 |
|---|---|---|
| 通常モード（Blob） | PC: 2-4 GB、スマホ: 500MB-1GB | RAM 依存 |
| 大容量モード（StreamSaver） | 数 GB〜数十 GB | ディスク容量次第、HTTPS必須 |
| SHA-256 検証あり | 500 MB | これを超えると検証はスキップ |

</details>

<details>
<summary><b>Q. 受信中に通信が切れたらどうなる？</b></summary>

A. v4 から自動で最大 **3 回まで再接続**を試みます（1.5秒・3秒・4.5秒の指数バックオフ）。受信済みのチャンク番号を記録しているため、「チャンク N 番から再開」で進捗を無駄にせず受信を続行します。

ただし、受信側のタブを閉じると受信済みチャンクは失われるため、最初からやり直しになります（タブを閉じても続けられる Resume はロードマップに掲載）。

</details>

<details>
<summary><b>Q. 双方向モード中にどちらかがタブを閉じたら？</b></summary>

A. 接続は終了し、双方向モードも解除されます。再度ファイルを送りたい場合は、最初からリンク発行をやり直す必要があります。これはサーバーレスである代償ですが、**両者がタブを閉じた瞬間にすべてのデータと鍵が破棄される**というプライバシー上のメリットでもあります。

</details>

<details>
<summary><b>Q. 受信者が複数いる場合は？</b></summary>

A. 現状は1対1での共有を想定しています。**2人目以降の接続は自動拒否**され、警告トーストが表示されます。複数の受信者に同じファイルを送りたい場合は、各受信者ごとに別々の共有リンクを生成してください。

</details>

<details>
<summary><b>Q. スマホで送信中、画面ロックされても大丈夫？</b></summary>

A. Screen Wake Lock API により、共有中は自動的に画面ロックが抑止されます。ただし iOS は 16.4 以降の対応です。古い iOS の場合は、設定で「自動ロック」を「なし」に変更することをおすすめします。

また、ZIP 圧縮やハッシュ計算など重い処理の**最初から** Wake Lock が取得されるよう設計されています（v4.1 で改善）。

</details>

<details>
<summary><b>Q. なぜタブを閉じると共有が止まる？</b></summary>

A. ファイルはあなたのブラウザのメモリに保持されており、WebRTC 接続もあなたのブラウザが終端しています。タブを閉じると両方が失われるため、共有が継続できなくなります。これは**サーバーレスである代償**でもあり、**プライバシー上のメリット**でもあります（サーバーに残るデータが一切ない）。

</details>

<details>
<summary><b>Q. 大容量モードを使う場合の注意点は？</b></summary>

A. 大容量モード（StreamSaver）は以下の制約があります:

- ⚠️ **HTTPS 環境必須**（HTTP / file:// では動作しません）
- ⚠️ **受信側ブラウザがポップアップをブロックしていない**こと
- ⚠️ **整合性検証（SHA-256）はスキップ**されます（ディスクに直書きするため再読み込み不可）

大容量ファイル（数GB以上）を扱う場合のみ ON にしてください。500MB 以下のファイルは通常モードのほうが整合性検証も含めて安全です。

</details>

<details>
<summary><b>Q. ファイル名に日本語や特殊文字を使っても大丈夫？</b></summary>

A. はい、問題ありません。ファイル名は UTF-8 で扱われ、表示時には HTML エスケープされるため XSS の心配もありません。ZIP 化される場合もファイル名は保持されます。

</details>
